# API Key 設定功能手動測試指引

本文件說明如何手動驗證 Etherscan 及 Tronscan API Key 設定功能，適用於功能驗收、回歸測試及部署後確認。

---

## 測試環境準備

在開始測試前，請確認以下前提條件均已滿足：

1. 系統已成功部署並可透過瀏覽器存取
2. 已準備兩組測試帳號：**管理員帳號**（role = admin）與**一般員警帳號**（role = user）
3. 備妥有效的 Etherscan API Key（可至 [https://etherscan.io/apis](https://etherscan.io/apis) 申請免費 Key）
4. 備妥有效的 Tronscan API Key（可至 [https://docs.tronscan.org/](https://docs.tronscan.org/) 申請）
5. 備妥一組無效的測試 Key（例如：`INVALID_KEY_FOR_TESTING`）

---

## 第一部分：系統設定頁面存取控制

### TC-UI-01：管理員可正常進入設定頁面

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 以管理員帳號登入系統 | 成功登入，顯示警員端首頁 |
| 2 | 點擊左側側邊欄「系統設定」 | 頁面導向 `/settings` |
| 3 | 確認頁面內容 | 顯示「系統設定」標題，可見「Etherscan API Key」與「Tronscan API Key」兩個設定卡片 |
| 4 | 確認每個卡片的初始狀態 | 若尚未設定，顯示「尚未設定」標籤；若已設定，顯示遮罩值（如 `ABCD****5678`） |

### TC-UI-02：一般員警無法看到設定操作按鈕

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 以一般員警帳號登入系統 | 成功登入 |
| 2 | 點擊左側側邊欄「系統設定」 | 頁面導向 `/settings` |
| 3 | 確認頁面內容 | 僅顯示 API Key 的設定狀態（已設定 / 未設定），**不顯示**「設定 API Key」或「清除」按鈕 |

## 第二部分：設定 Etherscan API Key

### TC-ETH-01：管理員成功設定有效的 Etherscan API Key

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 以管理員帳號進入「系統設定」頁面 | 頁面正常顯示 |
| 2 | 在「Etherscan API Key」卡片點擊「設定 API Key」 | 出現輸入框 |
| 3 | 貼上有效的 Etherscan API Key | 輸入框顯示輸入的內容 |
| 4 | 點擊「儲存」按鈕（或按 Enter） | 顯示成功提示訊息「Etherscan API Key 已更新」 |
| 5 | 確認卡片狀態更新 | 卡片顯示遮罩值（前 4 碼 + `****` + 後 4 碼），標籤由「尚未設定」變為「已設定」 |
| 6 | 重新整理頁面 | 設定持久保存，仍顯示遮罩值與「已設定」標籤 |

### TC-ETH-02：設定後遮罩格式驗證

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 設定一組已知的 Key，例如 `ABCD1234EFGH5678IJKL` | 儲存成功 |
| 2 | 確認顯示的遮罩值 | 應顯示 `ABCD****IJKL`（前 4 碼 + `****` + 後 4 碼） |
| 3 | 確認完整 Key 未顯示在頁面任何位置 | 頁面原始碼中不應出現完整 Key 字串 |

### TC-ETH-03：設定空值應被拒絕

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 以管理員帳號進入「系統設定」頁面 | 頁面正常顯示 |
| 2 | 點擊「設定 API Key」，不輸入任何內容直接點擊「儲存」 | 「儲存」按鈕應為禁用狀態，或顯示驗證錯誤訊息，不允許儲存空值 |

## 第三部分：設定 Tronscan API Key

### TC-TRON-01：管理員成功設定有效的 Tronscan API Key

操作步驟與 TC-ETH-01 相同，將「Etherscan」替換為「Tronscan」。

### TC-TRON-02：兩組 Key 可獨立設定

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 設定 Etherscan API Key | 成功，顯示遮罩值 |
| 2 | 設定 Tronscan API Key | 成功，顯示遮罩值 |
| 3 | 確認兩個卡片均顯示「已設定」 | 兩組 Key 獨立儲存，互不影響 |
| 4 | 清除 Etherscan API Key | Etherscan 卡片顯示「尚未設定」，Tronscan 卡片仍顯示「已設定」 |

---

## 第四部分：清除 API Key

### TC-CLR-01：管理員成功清除 API Key

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 確認 Etherscan API Key 已設定（顯示遮罩值） | 卡片顯示「已設定」 |
| 2 | 點擊「清除」按鈕 | 出現確認對話框（若有）或直接清除 |
| 3 | 確認清除 | 顯示成功提示，卡片恢復「尚未設定」狀態 |
| 4 | 重新整理頁面 | 清除狀態持久保存 |

### TC-CLR-02：一般員警無法清除 API Key

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 以一般員警帳號進入「系統設定」頁面 | 頁面正常顯示 |
| 2 | 確認「清除」按鈕不存在 | 頁面上不應顯示任何清除或修改按鈕 |

## 第五部分：API Key 對錢包查詢的影響

### TC-WLT-01：設定有效 Key 後錢包查詢正常運作

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 確認 Etherscan API Key 已設定 | 設定頁面顯示「已設定」 |
| 2 | 進入任一案件的「錢包分析」Tab | 頁面正常顯示 |
| 3 | 輸入一個已知的以太坊錢包地址（例如：`0x742d35Cc6634C0532925a3b844Bc454e4438f44e`）並點擊查詢 | 系統成功回傳錢包資料（地址、餘額、交易次數等），不顯示 API 速率限制錯誤 |

### TC-WLT-02：設定有效 Tronscan Key 後 TRON 錢包查詢正常運作

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 確認 Tronscan API Key 已設定 | 設定頁面顯示「已設定」 |
| 2 | 進入任一案件的「錢包分析」Tab | 頁面正常顯示 |
| 3 | 輸入一個已知的 TRON 錢包地址（例如：`TJDENsfBJs4RFETt1X1W8wMDc8M5XnJhCe`）並點擊查詢 | 系統成功回傳 TRON 錢包資料及 TRC-20 交易記錄 |

### TC-WLT-03：未設定 Key 時錢包查詢仍可運作（使用公開 API）

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 清除 Etherscan API Key | 設定頁面顯示「尚未設定」 |
| 2 | 進行以太坊錢包查詢 | 系統仍可查詢（使用公開 API，速率限制較低），不顯示系統錯誤，但可能因速率限制回傳部分資料 |

## 第六部分：更換 API Key

### TC-UPD-01：管理員可更換已設定的 API Key

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 確認 Etherscan API Key 已設定 | 顯示遮罩值 |
| 2 | 點擊「設定 API Key」（更換） | 出現輸入框 |
| 3 | 輸入新的 API Key | 輸入框顯示新 Key |
| 4 | 點擊「儲存」 | 成功提示，遮罩值更新為新 Key 的遮罩格式 |
| 5 | 進行錢包查詢確認新 Key 生效 | 查詢使用新 Key，舊 Key 立即失效 |

## 第七部分：邊界條件測試

### TC-EDGE-01：極短 Key（≤ 8 字元）的遮罩顯示

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 設定一個短 Key，例如 `ABCD` | 儲存成功 |
| 2 | 確認遮罩顯示 | 應顯示 `****`（不顯示任何原始字元） |

### TC-EDGE-02：極長 Key（接近 128 字元限制）

| 步驟 | 操作 | 預期結果 |
|------|------|----------|
| 1 | 設定一個 128 字元的 Key | 儲存成功 |
| 2 | 設定一個 129 字元的 Key | 系統拒絕，顯示驗證錯誤 |

## 自動化測試執行

除手動測試外，系統已內建 Vitest 自動化測試，可透過以下指令執行：

```bash
cd /home/ubuntu/case_filing_assist
pnpm test
```

測試涵蓋範圍如下表所示：

| 測試檔案 | 測試數量 | 涵蓋範圍 |
|----------|----------|----------|
| `server/settings.test.ts` | 27 個 | getAll、update、clear、getStatus、wallets.query API Key 優先順序 |
| `server/case_filing.test.ts` | 18 個 | 案件建立、OCR、情資分析、筆錄產生、settings CRUD |
| `server/auth.logout.test.ts` | 1 個 | 登出 Cookie 清除 |
| **合計** | **46 個** | **全系統核心功能** |

## 已知限制

Surya OCR 微服務在沙箱環境中出現 `SRE module mismatch` 錯誤，OCR 請求將自動 fallback 至 VLM 模式（視覺語言模型），功能仍可正常運作，但辨識速度可能較慢。此問題不影響 API Key 設定功能的驗證。
